- key props를 사용한 강제 리마운트 정리 (key 리마운트)
  - useEffect를 사용하지 않고, key props를 추가해 강제 리마운트하는 방법 및 여러 비교 정리
  - 사용 방법
    - 별다른 내용없이 key props 하나 추가하면 사용 가능
  - 사용해도 좋은 상황
    - 입력폼
    - Draft 메시지
    - 스크롤 상태
    - 애니메이션 초기화
    - 대상이 명확히 바뀌는 경우
      - userId, roomId, postId 등
      - 이전 상태를 유지하면 UX가 이상하고, 초기화가 자연스러운 경우
  - 주의해야 할 경우
    - 리스트가 아닌데 무조건 key 남용
    - key가 매 렌더마다 바뀌는 경우
      - ex) `Math.random()`
      - ex) `<Component key={Math.random()} />`
      - ex) `<Component key={Date.now()} />`
    - 유지되어야 할 state까지 날려버리는 경우
      - 입력 중 데이터
      - 복구 불가능한 UI 상태
      - 외부 SDK 연결
  - key로 리마운트 쓰는 대표 사례
    - 대상이 바뀌는 입력 폼
      - DM 대상
      - 댓글 대상
      - 수정 대상
      - `<Form key={entityId} />`
    - 스크롤 상태 초기화
      - `<MessageList key={roomId} />`
    - 애니메이션 다시 시작
      - `<Modal key={isOpen ? "open" : "closed"} />`
  - 이걸 잘 모르는 이유는?
    - 이유 1
      - 공식 문서에서도 90%는 리스트 예제
      - 강제 리마운트 트릭은 거의 나오지 않음
    - 이유 2
      - 실무에서도 useEffect 초기화로 우회하는 경우가 많음
      - key로 해결 가능한 걸 state/flag로 억지 관리함
    - 이유 3
      - React 팀은 기본 사용법 위주로 문서를 구성
      - key 리마운트는 고급 패턴에 가까움
      - 남용 위험 때문에 예제를 제한적으로만 제시함
      - 실무 + 경험 기반으로 퍼진 패턴
  - key에 대한 잘못된 정보
    - key를 map에서만 쓰는 것
      - key는 identity를 정의하는 수단
    - 위처럼 인식이 바쒸면 useEffect 초기화 내용이 많아지고, 상태가 꼬이며 조건 분기가 증가하게 됨
  - useEffect 초기화 vs key 리마운트 비교
    - useEffect 초기화
      - 코드량이 많아짐
      - 실수 가능성 높음
      - ref 초기화 수동
      - DOM 재생성하지 않음
      - 의도 표현 약함
    - key 리마운트
      - 코드량 거의 없음
      - 실수 가능성 낮음
      - ref 초기화 자동
      - DOM 재생성함
      - 의도 표현 강함
        - 새 컴포넌트라는 의도를 가장 명확히 표현하는 게 key
  - 그외 내용
    - key로 언마운트/마운트하는 것은 의도적으로 의미 있는 기준으로 사용하면 문제가 없으며 오히려 권장됨
    - 의미 있는 entity key는 권장되는 구조이며, 여러 경우에서 useEffect보다 key가 더 안전하고 명확함
    - 예로 채팅방 이동 - 페이지 이동 - 컴포넌트 언마운트 - state 초기화 이런 구조는 언마운트가 필요한 상황 즉, 라우터 전환이라는 생각이 박히게 되어있어서 같은 공간에서 대상만 바뀌는 경우를 "언마운트 대상"으로 인식하기 어려움
      - 기본적으로 컴포넌트가 공유된 상태에서 내용만 바뀌는 경우 이 때가 바로 key 리마운트 패턴의 사용 조건
      - key 리마운트는 공간(UI 위치)은 동일하고, 대상만 바뀌며 이전 state가 남아있으면 UX가 이상한 경우 사용
      - 내 경우 공간은 사용자 프로필, 대상은 보여지는 사용자 프로필 정보, 상태는 사용자 프로필에서 입력한 채팅 메시지 내용과 입력창 높이 그리고 스크롤이 해당됨
      - 위 상황은 사용자 프로필 자체가 계속 떠 있고, 내부 내용만 바뀌는 구조로 라우트 전환이 아닌 entity 전환이므로 key 리마운트를 사용하기에 적절
      - 반대로 채팅방 이동 같은 상황은 라우트 변경하고 컴포넌트 언마운트, state 자동 초기화를 하기 때문에 key 리마운트를 사용해도 의미가 없음
      - 또 다른 상황으로 조건부 렌더링으로 인해 꺼졌다 켜지면 언마운트, 마운트되기 때문에 라이프 사이클이 보장되어서 key 리마운트가 의미 없음
      - 언마운트되는 구조의 경우 key로 언마운트/마운트할 필요도 의미도 없게 됨
    - useEffect는 "같은 존재를 리셋"하는 도구, key는 "다른 존재임을 선언"하는 도구
  - 간단 정리
    - key는 `리스트 식별자`가 아닌, `컴포넌트의 존재 증명서`
    - 라우트 전환은 `공간 전환`이고, key 리마운트는 `존재 전환`

# 오늘 느낀 점

- 사용자 프로필 채팅 입력창 내에 주석 처리되어 사용하지 않는 스크롤 관련 내용을 모두 삭제하고, 사용자 프로필에서 새롭게 전달받은 프롭스 관련 타입 정의를 수정해 타입 에러 문제 해결, 그리고 사용자 세부 내용에 다이렉트 채팅방으로 이동하는 내용을 추가해 버튼 클릭 시에 이동할 수 있도록 구성을 해주었으며, 아바타 관련 내용이 보여지도록 구성을 해주었다. 추가로 key로 리마운트하는 내용도 조금 자세하게 정리를 해주었는데, 앞으로 저 내용은 사용할 상황에서는 최대한 사용하는 방향으로 구성을 해볼 생각이다. 사용자 프로필 세부 내용에서 보여지는 모달창 1/3, 2/3 또는 반으로 나눠서 왼쪽에는 사용자 아바타 관련 내용과 닉네임 그리고 메시지 버튼이 보여지도록 구성할 생각이며, 오른쪽에는 같이 아는 친구와 같이 참여한 채팅방 목록 내용이 출력되도록 구성할 생각인데, 우선 전체적인 틀부터 다시 잡은 다음 왼쪽에 보여지는 아바타 구조부터 만들어줄 생각이며, 그 다음 오른쪽에 보여지는 내용 구조를 수정할 생각이다.
